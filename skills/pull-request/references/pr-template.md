# Pull Request 描述範本

這是 Pull Request 描述的標準格式範本。

## 標準格式

```markdown
### 摘要
[一句話總結此 PR 的主要目的，例如：「實作使用者登入功能並整合 JWT 驗證」]

### 修改內容
- 變更點一：描述具體的修改內容
- 變更點二：描述具體的修改內容
- 變更點三：描述具體的修改內容

### ⚠️ 風險評估
[評估此 PR 是否有破壞性變更、需要特別注意的地方]

常見風險類型：
- 資料庫變更（migration、schema 修改）
- API 變更（endpoint 修改、參數變更、回應格式變更）
- 設定檔變更（環境變數、設定參數）
- 相依性更新（套件版本升級）
- 效能影響（查詢效能、記憶體使用）

若無風險，可註明：「無破壞性變更」

### 備註
[其他需要審查者注意的地方]

可包含：
- 測試方式或測試建議
- 部署注意事項
- 相關的 Issue 或 PR 連結
- 截圖或示意圖
- 待辦事項或後續工作
```

## 範例 1：新功能

```markdown
### 摘要
實作使用者登入功能，整合 JWT 驗證機制並新增 session 管理

### 修改內容
- 新增 `/api/auth/login` 和 `/api/auth/logout` API 端點
- 實作 JWT token 產生與驗證中介層
- 新增 User 資料表與相關的資料庫 migration
- 實作登入表單 UI 元件與驗證邏輯
- 新增登入相關的單元測試與整合測試

### ⚠️ 風險評估
- **資料庫變更**：需要執行 migration 建立 users 資料表
- **環境變數**：需要在 `.env` 中設定 `JWT_SECRET` 和 `JWT_EXPIRES_IN`
- **相依性**：新增 `jsonwebtoken` 套件

### 備註
- 測試帳號：`test@example.com` / `password123`
- 部署前請確認環境變數已正確設定
- 相關 Issue: #123
```

## 範例 2：Bug 修復

```markdown
### 摘要
修復使用者登出後 session 未正確清除的問題

### 修改內容
- 修正 logout API 的 session 清除邏輯
- 更新前端登出後的狀態重置流程
- 新增登出流程的測試案例

### ⚠️ 風險評估
無破壞性變更，僅修復現有功能的 bug

### 備註
- 已在 staging 環境測試確認
- 修復 Issue #456
```

## 範例 3：重構

```markdown
### 摘要
重構資料庫查詢邏輯，將重複的查詢抽取為共用函式並優化查詢效能

### 修改內容
- 將 User、Order、Product 的查詢邏輯重構為可重用的 Repository 模式
- 優化 N+1 查詢問題，使用 eager loading
- 新增查詢結果的快取機制
- 更新相關測試以符合新的架構

### ⚠️ 風險評估
- **架構變更**：從直接查詢改為 Repository 模式，但對外 API 保持不變
- **效能影響**：預期查詢效能提升約 40%（根據 benchmark 測試）
- **快取機制**：新增 Redis 快取，需確認 Redis 連線設定正確

### 備註
- 已執行完整的迴歸測試，所有測試通過
- Benchmark 測試結果：平均查詢時間從 120ms 降至 70ms
- 相關文件已更新：`docs/architecture/repository-pattern.md`
```

## 範例 4：多個 Commits

當 PR 包含多個 commits 時，摘要應該總結整體目的，修改內容應該整合所有變更：

```markdown
### 摘要
完整實作訂單管理功能，包含建立、查詢、更新訂單狀態等完整流程

### 修改內容
- 新增 Order 資料表與相關 migration（commit: abc123）
- 實作訂單 CRUD API 端點（commit: def456）
- 新增訂單管理前端頁面（commit: ghi789）
- 實作訂單狀態變更通知功能（commit: jkl012）
- 新增訂單相關的測試案例（commit: mno345）

### ⚠️ 風險評估
- **資料庫變更**：新增 orders 資料表
- **API 新增**：新增 `/api/orders` 相關端點
- **外部服務**：整合 email 通知服務，需確認 SMTP 設定

### 備註
- 測試涵蓋率：92%
- API 文件：`docs/api/orders.md`
- 相關 Issue: #789, #790
```

## 編寫指南

### 摘要部分
- 使用一句話清楚說明此 PR 的核心目的
- 避免過於簡短（如「更新程式碼」）或過於冗長
- 好的摘要範例：
  - ✅ 「新增使用者個人檔案編輯功能並整合頭像上傳」
  - ✅ 「修復付款流程中的金額計算錯誤」
  - ✅ 「重構認證中介層以提升程式碼可維護性」
- 不好的摘要範例：
  - ❌ 「更新程式碼」（過於簡短）
  - ❌ 「修改了登入相關的程式碼」（不夠具體）

### 修改內容部分
- 以列點方式說明具體變更項目
- 著重於「做了什麼」而非「改了哪些檔案」
- 若 commits 較多，可以按照功能分類或標註 commit hash
- 好的修改內容範例：
  - ✅ 「新增商品搜尋 API，支援關鍵字、分類、價格區間篩選」
  - ✅ 「實作購物車前端元件，包含新增、刪除、數量調整功能」
- 不好的修改內容範例：
  - ❌ 「修改 user.js、auth.js、login.vue」（只列檔案名稱）
  - ❌ 「更新程式碼」（不夠具體）

### 風險評估部分
- 認真評估此 PR 的潛在風險
- 常見風險類型：
  - 資料庫 schema 變更
  - API 介面變更（breaking changes）
  - 環境變數或設定檔新增/修改
  - 套件版本升級
  - 效能影響
- 若真的沒有風險，明確標註「無破壞性變更」
- 不要為了填寫而隨意編造風險

### 備註部分
- 提供額外的上下文資訊
- 常見內容：
  - 測試方式或測試帳號
  - 部署注意事項
  - 相關 Issue 或其他 PR 的連結
  - 截圖或示意圖（若適用）
  - 待辦事項或後續工作
- 若無特別需要說明的內容，可以省略此部分
